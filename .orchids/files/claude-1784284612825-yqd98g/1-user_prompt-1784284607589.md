Fix 3 issues found while verifying the Day-1 zero-state and Risk Register navigation
1. Risk Register should NOT be empty on Day 1 — and it's stuck empty even after saving real data
generateZeroState() sets topRisks: [], but risk identification is a planning-stage output derivable purely from site conditions (PSD applicability, cooling MGD, genset count, MW) — exactly what generateConstructionData()'s existing riskGenerators logic already computes, with no dependency on accumulated progress. Change generateZeroState() to compute real topRisks using the same risk-generation logic as generateConstructionData() (only progress/cost/safety accumulation fields should be zero — risks, cost category budgets, and the vendor/milestone plan are planning artifacts that exist from day one, not accumulated actuals).

Also confirmed live: after generating a fresh site (empty risk register, as expected pre-fix) and then saving real metrics via "Update Metrics," the risk register still shows "No risks registered" on the next fetch. Trace mergeSavedData() — likely cause: the first zero-state's empty topRisks: [] array got persisted into the saved DB row, and since [] || x evaluates to [] in JS (empty arrays are truthy), a naive merge keeps picking the empty saved array forever instead of falling back to freshly-computed risks. Fix: topRisks should always come from a fresh generateConstructionData() call, never persisted/merged from the saved row (the "Update Metrics" form has no way to edit risks in the first place).

2. Contingency Budget shouldn't show as $0
Same over-zeroing issue — contingencyBudget: 0 in generateZeroState(). Contingency is set as a percentage of TDC at planning time, before any of it is spent, so it should show a real computed value (e.g. 5-10% of BAC) on Day 1, with only contingencyUsed at $0. Check the same issue on costCategories: [], pcoList: [], and milestoneDetails: [] — also planning artifacts that should exist from day one with $0/0% actuals, not be entirely empty. The rule: anything representing "the plan" should be populated; anything representing "what's actually happened so far" should be zero.

3. Risk Register click-to-navigate likely routes to a broken destination
RiskRegister's onNavigate is wired to setActiveTab?.(tab), the top-level app router prop (valid values: 'overview', 'intake', 'permits', 'construction', etc.). But riskTabMap maps risk keywords to values like 'schedule', 'vendor', 'hse' — the internal Construction Dashboard sub-tab values. Calling the top-level router with those will try to navigate to a /schedule route that doesn't exist.

Fix: make the internal <Tabs defaultValue="csuite"> controlled (const [activeSubTab, setActiveSubTab] = useState('csuite'), wire <Tabs value={activeSubTab} onValueChange={setActiveSubTab}>). onNavigate should call setActiveSubTab(tab) for internal destinations (schedule, vendor, hse, etc.), and only use the top-level setActiveTab for the genuinely cross-page case ('intake').
