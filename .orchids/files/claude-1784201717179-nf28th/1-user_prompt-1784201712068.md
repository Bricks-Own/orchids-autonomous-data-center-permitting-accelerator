Make the 4 permit detail pages (Air/Water/Building/Power) visually consistent and use real shadcn components
AirPermitAI.jsx, WaterPermitAI.jsx, BuildingPermitAI.jsx, PowerPermitAI.jsx (rendered inline from Permits.jsx when a permit type card is clicked) currently import zero shadcn components between them. Every banner, metric card, table, and expandable module list is hand-rolled raw HTML, and each file picked its own random accent color — Air mixes red/amber/green, Water is blue, Building is mostly primary/amber, Power is yellow — none tied to the app's actual lime/emerald theme. Status indicators use ASCII glyphs (V, X, ~, ▲, ▼) or emoji (💨💧🏗️⚡📋📅) instead of the Phosphor icons used everywhere else (SiteIntake.jsx's PERMIT_TYPE_CONFIG already has the correct icon for each: Wind/Drop/Buildings/Lightning — reuse those exact icons here).

Important: Air's file is ~750 lines with real, unique content (breach analysis, remediation matrix, BACT strategy) that Water/Building/Power don't have equivalents for. Don't strip or flatten that content to "match" the others — the goal is a shared visual/structural language, not identical feature depth. Each permit type keeps its own domain-specific sections; only the shared patterns (banner, metric cards, tables, module list) should become one consistent implementation.

Required changes
1. Extract shared components, used by all four files
Create these once (e.g. in a new permitOS/src/components/permit-detail/ folder or alongside the existing components) and import them into all four files instead of each file re-implementing its own version:

PermitBanner — the header banner at the top of each page (site name/permit type, key stats, status badges). One consistent Card layout, not four different colored boxes.
MetricCard — the small stat cards (Water/Building/Power's waterMetrics/buildingMetrics/metrics arrays all map to visually-almost-identical cards already — unify them into one Card-based component).
PermitModuleAccordion — the expandable "Modules — Deliverables" list (AIR_MODULES/WATER_MODULES/BUILDING_MODULES/POWER_MODULES, all rendered with the same hand-rolled expand/collapse useState + ▲/▼ pattern in every file). Replace with the shadcn Accordion component (check permitOS/src/components/ui/ for an existing accordion.jsx; if none exists, add it via the shadcn CLI matching the Sera preset) — one component, four call sites.
PathwayDeterminationCard — the pattern used in Air's "Permit Pathway Determination", Water's pathway list, Building's "Building Code Pathway", and Power's "Interconnection Pathway" — all four are the same shape (label / status / detail / citation), each currently hand-rolled slightly differently. One shared component.
2. Use the established per-permit-type color convention
DocumentFactory.jsx already solved this exact problem for the Documents tab — it has a CAT_STYLE map keyed by permit type (air → chart-1, water → chart-2, building → chart-3, power → chart-4) with the rule "background/border can use the accent color, but title text stays white/foreground." Apply the identical convention here, replacing every arbitrary blue-400/yellow-400/orange-400/violet-700/etc. class in all four files.

3. Replace raw <table> with shadcn Table components
Air's "Per-Pollutant Major Source Determination," "Remediation Action Matrix," and PTE comparison grid all use raw <table>/<thead>/<tbody> markup. Convert to Table/TableHeader/TableBody/TableRow/TableHead/TableCell.

4. Replace ASCII/emoji glyphs with Phosphor icons
Status glyphs V / X / ~ → CheckCircle / XCircle / WarningCircle.
Expand/collapse ▲ / ▼ → the shadcn Accordion's built-in chevron from step 1.
Cross-tab nav button emoji → the same Phosphor icons already defined in App.jsx's tabIcons map — reuse that exact mapping.
Checkmark ✓ in deliverables lists → Phosphor Check.
5. Standardize the empty state ("No Site Data Yet")
Standardize all four to one pattern: a Phosphor icon (not emoji), the same copy, and a real shadcn Button (variant="default") linking to Site Intake.

6. Every metric, status, and value must be derived from Site Intake data — not hardcoded
Confirmed concrete examples of hardcoded/static values that should be data-driven, in WaterPermitAI.jsx's "Water Permit Pathway Determination" list:

Construction Stormwater — CGP: status: 'Required' is hardcoded regardless of input, even though the trigger text itself says "Site disturbance ≥ 1 acre" — this should check inputs.siteAcres (or inputs.hasNewConstruction) and only show as required/applicable when the actual site data crosses that threshold.
SPCC — Fuel and Oil Storage: status: 'Required' is hardcoded for any gensetCount > 0, but the citation itself says the real regulatory trigger is "above-ground storage > 1,320 gal." The component has inputs.gensetCount available but never computes actual aggregate diesel storage from genset count/size to check it against the real threshold — it should.
Pretreatment / POTW Discharge: hardcoded status: 'Evaluate with Utility' regardless of inputs.dischargePathway — this is already captured as a real field on Site Intake (dischargePathway === 'POTW-Sanitary Sewer Connection'), so this item's applicability should key off that instead of always showing the same generic status.
Wetlands / Waters of the US: hardcoded status: 'Screening Required' — this one may legitimately not be computable from current inputs, which is fine, but it should be explicitly labeled as a standing recommendation rather than implying it was evaluated against this specific site.
Do a full sweep of all four files with this lens: any status, ok, badge, or displayed number that doesn't already trace back to a field on results or inputs needs to either (a) be made conditional on the actual entered/computed value, or (b) if it's genuinely generic regulatory guidance that can't be computed from current inputs (like BACT strategy technology descriptions, or citation/regulation text), that's fine to leave as reference content — just don't let it read as if it were computed specifically for this site when it wasn't.

What NOT to change
Don't remove or shorten Air's breach analysis, remediation matrix, or BACT strategy sections — they're real, unique content.
Don't remove Water's cooling tower water balance section, Building's building code pathway specifics, or Power's interconnection pathway specifics — same reasoning.
Don't change any of the existing underlying calculation logic (computeBuildingMetrics, computePowerMetrics, threshold analysis, calcPTE, etc.) — this is otherwise a display-layer restyle. The one exception is section 6 above: where a displayed status is currently hardcoded and should instead be computed from inputs/results, add that specific computation — don't touch anything else in the calculation layer.
