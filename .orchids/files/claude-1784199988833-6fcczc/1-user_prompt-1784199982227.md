Add permit-type selection to Site Planner and make it flow through to Site Intake
The gap
derivedPermitTypes in SiteIntake.jsx (~line 95-100) decides which permit types (air/water/building/power) a site needs based on four boolean flags in shared context: hasOnSiteGeneration, hasWaterUse, hasNewConstruction, hasGridInterconnection. These flags currently only get set on Site Intake's own Step 1 Q&A (PROJECT_QUESTIONS), and default to true if never touched.

SitePlanner.jsx never references any of these four flags anywhere. That means every site built in the Planner — regardless of what the user actually configures on the map or picks from the Scenario Explorer — silently assumes all four permit types are needed once it reaches Intake, and there is nothing on the Planner UI that shows the user which permit domains currently apply to their site.

What to build
1. Add a "Permit Requirements" section to Site Planner
Add a new Card to the left column of SitePlanner.jsx (near "Current Site Configuration"), using the exact same Q&A copy and toggle-card visual pattern already built in SiteIntake.jsx's Step 1 PROJECT_QUESTIONS (the four clickable cards with a checkmark badge in the corner when "Yes" — reuse the same questions verbatim so the language is identical in both places):

"Will this site include on-site power generation — turbines, generators, or other combustion equipment?" → hasOnSiteGeneration
"Will this site use or discharge water — cooling towers, process water, wastewater?" → hasWaterUse
"Is this new building construction, or a build-out inside an existing structure?" → hasNewConstruction
"Will this site need a new or upgraded grid interconnection?" → hasGridInterconnection
To avoid duplicating this config in two files, move PROJECT_QUESTIONS out of SiteIntake.jsx into a shared location (e.g. permitOS/src/data/permitData.js, alongside the other shared constants) and import it from both SiteIntake.jsx and SitePlanner.jsx.

Wire the toggles directly to setInputs the same way every other Planner control already does, so they live-update shared context immediately — no separate "apply" step needed for these.

2. Show which permit types are currently active, visually, before the user leaves the Planner
In the "Current Site Configuration" card (or directly above/below it), show four small Badge components — Air / Water / Building / Power — styled active (filled) or inactive (muted/outline) based on the current flag values. This is the "at a glance" confirmation that Site Intake will pick up the same permit set the user sees here.

3. Set realistic flags on the 4 Scenario Explorer presets
Currently none of the Edge Site / Colocation Expansion / Hyperscale Campus / CA Nonattainment preset params objects set these four flags at all — they fall through to the default true. Add explicit hasOnSiteGeneration, hasWaterUse, hasNewConstruction, hasGridInterconnection values to each preset's params, based on what's actually in that preset's equipment profile (e.g. if a preset's params includes turbines/gensetCount it should set hasOnSiteGeneration: true; if it includes buildingSqFt it should set hasNewConstruction: true; etc.) — the goal is that every preset explicitly declares its permit footprint rather than relying on an implicit default that happens to always be true.

4. Reflect the active permit types in the "sent to Intake" confirmation banner
navigateToIntake(summary) in SitePlanner.jsx currently passes a summary string like "Hyperscale Campus — 800MW" to the banner shown on Site Intake (showPlannerBanner in SiteIntake.jsx). Extend that summary to also list which permit types are active, e.g. "Hyperscale Campus — 800MW — Air, Water, Building, Power", so the banner on Intake confirms both the scenario and the permit scope that carried over.

5. Verify the downstream effects, don't just implement blind
After wiring this up, confirm:

Toggling a flag off in the Planner (e.g. hasWaterUse → No) and pushing to Intake correctly hides the "Water-Specific Parameters" section on Step 2 and the "Water Permit" card on Step 3 (this gating logic already exists in SiteIntake.jsx via permitTypes.includes(...) — it should just work once the flags are correctly set from the Planner, but confirm it live).
The same toggle-off correctly removes that permit type from Permits.jsx and from the Document Factory's generated document list (showAir/showWater/showBuilding/showPower in DocumentFactory.jsx already derive from these same flags — confirm the chain holds end-to-end, Planner → Intake → Permits → Documents).
What NOT to change
Don't touch the Site Intake Step 1 Q&A itself beyond importing the now-shared PROJECT_QUESTIONS — its behavior should stay identical.
Don't change the navigateToIntake / fromPlanner handoff mechanics beyond the summary string — that logic was just fixed and verified working.
