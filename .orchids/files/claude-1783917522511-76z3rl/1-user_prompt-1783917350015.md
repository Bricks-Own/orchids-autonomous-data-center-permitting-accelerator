ITEM 0 — CRITICAL BUG: 3D Site View renders a black box (WebGL context error)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

On the deployed production build, opening Construction Platform → "3D Site View" renders nothing: the tab chrome, timeline slider, and legend appear, but the canvas stays at its default 300×150 size and every pixel is black. The console throws, four times, on a clean page load:

  THREE.WebGLRenderer: A WebGL context could not be created. Reason: Canvas has an existing context of a different type

Where to look: src/components/SiteView3D.jsx renders a @react-three/fiber <Canvas> (line ~317, with gl={{ antialias: true, alpha: false }}), mounted conditionally from src/components/ConstructionDashboard.jsx line ~877 ({subTab === '3dview' && ... <SiteView3D data={data} />}). The error means Three.js is trying to grab a 'webgl' context on a <canvas> element that already has a context of a different type — the classic causes are: (a) the component unmounting/remounting and R3F reusing a DOM canvas that something else (another renderer instance, a 2d context call, or a stale previous Canvas) already claimed; (b) double-mounting from React StrictMode creating two renderer instances against the same canvas; (c) the canvas being created, a context lost/acquired, then a second Canvas mount racing the first one's cleanup when the user switches sub-tabs quickly or the parent re-renders (note ConstructionDashboard re-fetches and re-renders on every [inputs, results] change — line ~240 — which can remount the 3D view repeatedly).

Fix requirements:
- The 3D view must render reliably on first open, after switching away to another sub-tab and back, and after the parent dashboard refetches data. Test all three paths explicitly.
- Give the Canvas a stable identity across parent re-renders (e.g. key it stably, memoize the wrapper so data refetches don't remount the renderer, and ensure the previous WebGL context is disposed before a new Canvas mounts — R3F's unmount cleanup must be allowed to finish).
- Also fix the canvas sizing: it must fill its container (not 300×150). Wrap it in an explicitly sized container div.
- Verify by loading the page fresh (no cached state), navigating Construction Platform → 3D Site View, confirming the model actually renders (non-black pixels), then switching to another sub-tab and back twice, and confirming zero console errors throughout.
