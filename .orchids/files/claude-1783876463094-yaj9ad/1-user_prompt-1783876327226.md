BUG: Site Planner has 6 separate places that can change the site's location/parameters, and they write to the shared inputs object inconsistently — none of them keep Address/County in sync with State/Lat/Lon, and some of them (Scenario Explorer cards) sync a lot of fields but still miss address/county. Net effect: whichever path the user takes, Site Intake can end up showing mismatched location data (confirmed repro: pick "Phoenix, AZ" preset → Site Intake shows State=Arizona, Lat/Lon=Phoenix, but Address/County are still whatever was there before, e.g. "Davidson County"/Nashville).

The 6 paths in src/components/SitePlanner.jsx that currently write to `inputs` independently:
1. Quick Location Presets (~line 382) — sets state, lat, lon, siteAcres
2. Map marker drag / map click → handleLatLonChange (~line 307) — sets lat, lon, and state only if reverse-geocoded from coordinates
3. Manual Lat/Lon text inputs (~line 422) — calls the same handleLatLonChange
4. "Send to Site Intake" button (~line 497) — sets state, lat, lon, siteAcres
5. Scenario Explorer cards: Edge Site, Colocation Expansion, Hyperscale Campus, CA Nonattainment (~line 579-644) — sets a large bundle of fields (turbines, mwPerTurbine, hours, brickSavings, gensetCount, gensetHours, datacenterMW, siteAcres, buildingSqFt, stories, occupancyType, fireSuppression, emergencyPowerConfig, powerSourceType, interconnectionVoltage, transformerCapacity, and for CA also state/lat/lon/nonAttainment flags)
6. Custom Scenario Builder's "Apply Parameters to Site Intake" button (~line 289, ScenarioTest component) — merges its local slider state back

None of these 6 touch `address` or `county`. Site Intake's own auto-fill of address/county from STATE_ADDRESS_DEFAULTS only runs inside its State `<select>`'s onChange handler (src/components/SiteIntake.jsx ~line 211), which none of the above 6 paths trigger.

FIX — do this properly, not with another one-off patch:
1. Create one shared helper (e.g. `applyLocation(setInputs, { state, lat, lon, acres, presetLabel })` in a shared utils file) that all 6 call sites use instead of hand-rolling their own `setInputs(...)` calls. This helper should look up the correct address/county for the location (reuse/extend the existing STATE_ADDRESS_DEFAULTS table in SiteIntake.jsx — move it to shared data if needed — and set address/county/lat/lon/state together, atomically, every time location changes from any source).
2. For the 8 Quick Location Presets and 4 Scenario Explorer cards specifically, since they represent named real-world locations (e.g. "Ashburn, VA (Data Center Alley)", "Dallas, TX (Hyperscale)"), give each one its own specific realistic site address/county instead of falling back to the generic state-level default — this is a demo, and a named preset showing a made-up-sounding generic address undercuts it.
3. 4. Respect the existing `addressFieldsTouched` guard in SiteIntake.jsx so this never overwrites an address the user typed by hand there.
