BUG: The "Send to Site Intake" button in Site Planner (src/components/SitePlanner.jsx, ~line 484-490) still produces wrong/generic location data, even after the recent applyLocation() fix. Two separate issues:

1. WRONG STATE from ambiguous bounding-box overlap. The button's onClick does:
   `const detected = getStateFromCoords(lat, lon); const stateToUse = detected || selectedState; applyLocation(setInputs, { state: stateToUse, lat, lon, acres: siteAcres });`
   getStateFromCoords() (defined ~line 8) checks lat/lon against STATE_BOUNDING_BOXES (src/data/permitData.js), which are simple rectangles that overlap between neighboring states. Confirmed repro: select the "Ashburn, VA (Data Center Alley)" preset (lat 39.0438, lon -77.4874), then click "Send to Site Intake" — Site Intake shows State: Maryland, not Virginia, because Maryland's box [37.9, 39.7, -79.5, -74.9] and Virginia's box [36.5, 39.5, -83.7, -75.1] both contain that coordinate, and Maryland happens to be listed first in the object so it wins the lookup. The button throws away the already-correct `selectedState` (which was properly set when the preset was clicked) in favor of this ambiguous guess.

2. GENERIC ADDRESS instead of the specific preset/scenario address. applyLocation() now supports `presetLabel` and `scenarioTitle` params to look up a specific realistic address (see PRESET_SPECIFIC_ADDRESSES and SCENARIO_CARD_ADDRESSES in src/utils/locationUtils.js) instead of falling back to the generic STATE_ADDRESS_DEFAULTS. The "Send to Site Intake" button never passes either param. Confirmed repro: select the "Dallas, TX (Hyperscale)" preset, click "Send to Site Intake" — Site Intake shows Address: "10001 Wallisville Rd, Houston, TX 77029" (the generic Texas fallback) instead of "8700 E McCommas Blvd, Dallas, TX 75241" (the actual Dallas preset address that's already defined and used correctly by the preset buttons themselves).

FIX:
1. Stop re-deriving state from coordinates in this button. `selectedState` in SitePlanner's local state is already the source of truth (it's set correctly by every preset click, map interaction, and scenario card). Just pass `state: selectedState` directly to applyLocation instead of calling getStateFromCoords() and preferring its result.
2. Track which preset (if any) is currently active — e.g. a `selectedPresetLabel` state variable set whenever a Quick Location Preset or location-bearing Scenario card is clicked, cleared when the user manually drags the map or types Lat/Lon — and pass it through to applyLocation as `presetLabel` (or `scenarioTitle`) when "Send to Site Intake" is clicked, so it resolves the same specific address the preset/scenario itself would have used, not the generic state fallback.
3. Separately, the underlying STATE_BOUNDING_BOXES overlap between Maryland and Virginia (and possibly other adjacent state pairs — worth a quick audit) is a latent bug that also affects the map-marker-drag and manual-Lat/Lon-entry paths, not just this button. It doesn't need a full geocoding API to fix — at minimum, tighten the box boundaries so they don't overlap in populated areas, or when multiple states match, prefer the one whose box center is closest to the clicked point rather than just taking the first object key that matches.

VERIFY after the fix: click each of the 8 Quick Location Presets and each of the 4 Scenario Explorer cards, then click "Send to Site Intake" for each — confirm State/County/Address/Lat/Lon all match the specific location that was selected, with no generic-fallback or misdetected-state cases. Pay special attention to Ashburn VA (MD/VA border) and Northern Virginia (also near MD/DC) since those are the ones that exposed the bounding-box overlap.
