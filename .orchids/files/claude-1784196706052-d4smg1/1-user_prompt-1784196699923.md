Site Intake handoff is broken and confusing
The bug, confirmed live
I reproduced this end-to-end in the running app (not just reading code):

On Site Intake, ran a full screening for the default site (8×25MW = 200MW). Got real results: Traditional ~18 months, Brick-Accelerated ~14 months, Baseline NOx 357.3 tpy, PSD Major.
Navigated to Site Planner, clicked the "Hyperscale Campus" scenario card's "Apply → Intake" button (16×50MW = 800MW, a completely different equipment profile).
Landed back on Site Intake, Step 3 (Review & Generate) — but it was still showing the old 200MW numbers: ~18 months / ~14 months / 357.3 tpy NOx. Nothing had recalculated.
The Generate My Permits button was disabled (because results was still non-null from the old run), and a Reset button had appeared — but there is nothing on screen telling the user this happened or that they need to click Reset before they can see numbers that match their new 800MW site.
So right now, changing your site on the Planner and pushing it to Intake silently shows stale, wrong numbers with no way to tell they're wrong, and the only way to fix it is to know to click a "Reset" button that gives no explanation of why it's there.

There's a second, related problem: if you push data from Site Planner to Intake on a fresh session (no results generated yet), you land on Step 1 — Project Setup, the generic Yes/No question cards. There is no visual confirmation that anything was actually carried over from the map/scenario you just configured — the user has to click through two more steps just to check whether it worked, which reads as "it didn't do anything."

Root cause
File: permitOS/src/components/SitePlanner.jsx

Three separate call sites push new parameters into the shared context and then navigate to /intake, but none of them clear the previously-generated results:

ScenarioTest's "Apply Parameters to Site Intake" button (~line 312-317)
The "Send to Site Intake" button (~line 512-517)
Each Scenario Explorer card's "Apply → Intake" button (~line 658-683)
All three only call setInputs(...) / applyLocation(setInputs, ...) and then setActiveTab('intake'). SitePlanner never touches results at all — it only pulls { inputs, setInputs } from usePermitData() (line 324).

File: permitOS/src/components/SiteIntake.jsx

const [step, setStep] = useState(() => (results !== null ? 3 : 1));
This only runs once, on mount. Combined with the bug above:

If results already exists from a prior run, arriving from Site Planner always lands on Step 3, displaying results computed from the old inputs, visually indistinguishable from a correct, up-to-date result.
If results doesn't exist yet, it lands on Step 1, which shows no trace of the data that was just configured on the map.
There is currently no signal anywhere — no banner, no toast, no field highlight — telling the user "these parameters came from Site Planner" or "your results are out of date, regenerate."

Required fix
1. Clear stale results when Site Planner pushes new parameters
In SitePlanner.jsx, pull setResults from usePermitData() alongside inputs/setInputs:

const { inputs, setInputs, setResults } = usePermitData();
Then call setResults(null) in all three places that currently push params and navigate to Intake, right before navigating:

The ScenarioTest "Apply Parameters to Site Intake" button handler
The "Send to Site Intake" button handler
Each Scenario Explorer card's "Apply → Intake" click handler (both the card-body click path and the small inline button's onClick, since the code currently duplicates this logic in two places — see lines ~596-617 and ~658-683)
This guarantees the Review step never shows numbers that don't match the currently-loaded site.

2. Land on Step 2 (Site Details), not Step 1, when arriving from Site Planner — and show a confirmation banner
When results are null and the user just came from Site Planner, don't dump them on the generic Project Setup Q&A (they didn't come here to answer "will this have on-site generation" — they already built the site on the map). Take them straight to Step 2 — Site Details, where they can see the exact fields that were just populated (turbines, MW, location, building/power params) before running screening.

Implementation: in SitePlanner.jsx, use useNavigate() directly (import from react-router-dom) for these three CTAs instead of the setActiveTab('intake') wrapper, and pass router state:

import { useNavigate } from 'react-router-dom';
// ...
const navigate = useNavigate();
// ... in each handler, replace setActiveTab('intake') with:
navigate('/intake', { state: { fromPlanner: true, summary: '<short description of what was applied, e.g. "Hyperscale Campus — 800MW"' } });
In SiteIntake.jsx, read that state to decide the initial step and show a banner:

import { useLocation } from 'react-router-dom';
// ...
const location = useLocation();
const fromPlanner = location.state?.fromPlanner;
const [step, setStep] = useState(() => {
  if (results !== null) return 3;
  if (fromPlanner) return 2;
  return 1;
});
At the top of Step 2 (or globally, above the Stepper), render a dismissible confirmation banner when fromPlanner is true, e.g.:

✓ Parameters loaded from Site Planner — {location.state.summary}. Review below, then continue to generate permits.

This directly answers "did it actually work" the instant the page loads, instead of making the user hunt for evidence.
