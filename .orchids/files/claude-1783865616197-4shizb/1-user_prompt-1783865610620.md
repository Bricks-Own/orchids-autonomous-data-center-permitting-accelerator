Fix the following issues found in a QA audit, one at a time, verify each before moving to the next:

1. Construction Platform C-Suite crash (critical) — ConstructionDashboard.jsx throws ReferenceError: contingency is not defined and blanks the screen. Root cause: line 320 renames the contingency data to cont (const cont = data.contingency || {}), but line 451 still references the old name contingency (${(contingency?.budget / 1e6).toFixed(1)}M). Fix by changing line 451 to use cont?.budget instead of contingency?.budget. Also check the whole file for any other leftover bare contingency/evm/safety/quality/schedule/chg references that should be cont/etc. — this exact bug pattern (rename refactor leaving old variable names behind) has caused three prior crashes in this codebase (Building Permitting, Power Permitting, Executive Summary), so audit this file thoroughly rather than just fixing the one line.

2. Same file, escaped template literal bug — around line 438, a progress bar width is written as \${Math.min(...)}% with a literal backslash before the $, so it renders as a literal string instead of computing a percentage — the bar never visually fills. Remove the backslash so it's a real template literal: ${Math.min(...)}%.

3. Construction Platform hardcodes one site ID for everyone — ConstructionDashboard.jsx (around lines 235, 264) calls the construction API with a hardcoded BIGWATT-SITEA-001 regardless of what site the user actually created in Site Intake. Fix by using the real site's ID/name from the actual site data instead of the hardcoded string.

4. Construction metrics use Math.random() and change on every refresh — server/constructionData.js (around lines 469, 533, 555, 625-686) uses Math.random() inside metric calculations, so identical inputs produce different KPI values (e.g. CPI showing 1.049 then 1.027 on consecutive calls with the same data). Remove the randomness — these should be deterministic calculations based on the actual site inputs, consistent every time for the same input.

5. AIR-005 document still shows a literal unfilled placeholder — [State Air Agency — Permit Section] prints verbatim in the generated document. Root cause: formConverter.js line 28 only has a resolver for the hyphen version ('[State Air Agency - Permit Section]'), but documentGenerator.js line 359 uses an em-dash version ([State Air Agency — Permit Section]). Fix by adding a resolver entry for the em-dash version too (or standardizing the template to use whichever the resolver expects).

6. Threshold mismatch between two files for interconnection voltage — SiteIntake.jsx (~line 113) uses > 200 to decide 138kV vs 69kV, but buildPowerCalc.js (~line 66) uses >= 200 for the same decision. A site at exactly 200 MW gets a different answer depending which code path runs. Fix by making both use the same comparison (recommend >= 200, matching buildPowerCalc.js).

Test each fix by: (1) opening Construction Platform → C-Suite Dashboard after running screening, confirm no blank screen; (2) confirming the contingency progress bar visually fills; (3) creating a site with a custom name and confirming Construction Platform reflects it, not BigWatt; (4) calling the construction data twice with identical inputs and confirming identical output; (5) opening AIR-005 in Document Factory and confirming no bracketed placeholder text; (6) testing a site at exactly 200 MW and confirming consistent voltage across tabs.
