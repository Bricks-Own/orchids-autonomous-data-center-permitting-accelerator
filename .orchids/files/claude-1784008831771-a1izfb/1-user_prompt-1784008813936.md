MAJOR REDESIGN — Permit-type-driven intake flow (replaces the current "one giant form + hunt through 15 tabs" flow)

CONTEXT: Right now, Site Intake (src/components/SiteIntake.jsx) is a single form that asks every user for every field regardless of which permits they actually need — building code fields, power/interconnection fields, air permit fields, and water permit fields are all shown unconditionally. "Run Full Permit Screening" then computes results for all permit types whether the user asked for them or not. After that, the user is dropped into a 15-tab nav bar (src/components/Header.jsx, the `tabs` array) and has to manually click around to find their own results. This is confusing and doesn't match how a real user thinks about the problem: they know upfront which permits they're pursuing (Air? Water? Building? Power/Interconnection? — often more than one), and they want to enter only what's relevant, then go straight to their results.

REDESIGN THE FLOW INTO THREE STEPS:

STEP 1 — Permit type selection (new, first thing shown on Site Intake, before any other fields).
A checklist/multi-select: "Which permits are you pursuing?" with options: Air Permit, Water Permit, Building Permit, Power / Interconnection. Multi-select — most sites need several. Store the selection on the shared inputs object (e.g. inputs.permitTypesNeeded = ['air','water',...]) so it persists across the session and can be edited later (don't force the user through this every time — let them revisit/change it from Site Intake).

STEP 2 — Dynamic form scoped to the selection.
Site Intake keeps a small set of universal fields everyone fills regardless of selection (Site Name, Client, State, County, Address, Lat/Lon, Site Acreage — the identity/location fields, plus the core generation fields that basically everything downstream depends on: Turbines, MW per Turbine, Operating Hours, Brick Savings — these drive PTE calc for almost every permit type). Beyond that:
  - Only show the AIR-specific fields (turbine/engine type, stack height, nearest receptor distance, nonattainment flags) if 'air' is selected.
  - Only show WATER-specific fields (cooling MGD, blowdown %, water MGD, discharge pathway — see the new field below) if 'water' is selected.
  - Only show BUILDING-specific fields (building sqft, stories, occupancy type, fire suppression, emergency power config) if 'building' is selected.
  - Only show POWER-specific fields (interconnection voltage, transformer capacity, power source type, genset count/HP/hours — see the fuel-type fix below) if 'power' is selected.
Keep every field's underlying data key exactly as-is (buildingSqFt, coolingMGD, etc.) — this is a visibility/grouping change in the UI, NOT a rename of any field the calculation engines (src/utils/calculations.js, buildPowerCalc.js, documentGenerator.js) depend on. Do not touch those calculation files.

STEP 3 — One action, one destination.
Replace "Run Full Permit Screening" with a single "Generate My Permits" button. It runs calcPTE/apiPTE as today, but the results view and downstream nav should now be scoped to what was actually selected:
  - After generating, route the user to a results landing view that shows ONLY the sections for their selected permit types (not a blank dashboard requiring them to go find their own results).
  - PRUNE THE NAV BAR: in src/components/Header.jsx, the `tabs` array currently always shows all of Air Permit AI, Water Permit AI, Building Permitting, Power Permitting regardless of what the user needs. Make those four tabs conditionally visible based on inputs.permitTypesNeeded — if the user didn't select Water, don't show the Water Permit AI tab at all. This needs `inputs` (or at least the permitTypesNeeded array) passed into Header, which it doesn't currently receive — thread it through from App.jsx.
  - Cross-cutting tools stay ALWAYS visible regardless of permit selection, since they're not permit-type-specific: Platform Overview, Executive Summary, Site Planner, Site Intake, Milestone Timeline, Document Factory, Digital Twin, Compliance OS, Regulator Copilot, Knowledge Hub.
  - CONSTRUCTION PLATFORM STAYS SEPARATE — do not gate it behind the permit-type checklist and do not treat it as a 5th permit-type option. It's a different phase of the project (tracking an active build, not applying for a permit) with its own independent inputs (budget baseline, vendors, safety) that have nothing to do with permit applicability. It should remain always-visible in the nav, exactly as it is today.

TWO REAL DATA-MODEL BUGS TO FIX AS PART OF THIS (found during a field audit — not cosmetic, these produce wrong or placeholder output today):

BUG A — "Turbine / Engine Type" dropdown incorrectly conflates primary generation with emergency backup. In src/components/SiteIntake.jsx line ~222 and src/data/permitData.js, the dropdown is built from `Object.keys(NOX_EMISSION_FACTORS)`, which mixes real primary turbine options ('Gas Turbine (DLN, modern)', 'Gas Turbine (standard combustion)', 'Gas Turbine (older frame, uncontrolled)') together with emergency backup options ('Emergency Diesel Genset', 'Emergency Gas Genset') in ONE selector meant to represent the site's primary generation. A user can nonsensically select "Emergency Diesel Genset" as their primary turbine. Fix: split this into two separate concepts —
  1. "Primary Turbine Type" — only the three real turbine options.
  2. A new "Genset Fuel Type" selector (Diesel / Natural Gas) shown alongside the existing genset count/HP/hours fields (only when 'power' or the genset section is relevant) — this is currently MISSING entirely from the form.
  Then in src/utils/calculations.js (~line 29, the `gensetEfs` object), make the genset emission factors conditional on the new genset fuel type field instead of the current unconditional hardcoded diesel AP-42 factors — use the existing 'Emergency Diesel Genset' / 'Emergency Gas Genset' entries in NOX_EMISSION_FACTORS / CO_EMISSION_FACTORS (src/data/permitData.js) as the two factor sets, keyed by the new field, instead of leaving 'Emergency Gas Genset' as dead/unused data. Also update src/utils/documentGenerator.js genAir3_FuelTankInventory (~line 153), which currently hardcodes "DIESEL FUEL INVENTORY" and diesel gallon math unconditionally — it should branch on genset fuel type (a natural-gas genset doesn't need an above-ground diesel day tank / SPCC analysis the same way).

BUG B — Water discharge pathway is never asked for, but a generated document claims to need it. src/utils/formConverter.js line ~35 and src/utils/documentGenerator.js line ~1184 (the Water Balance document, WATER-002) print the literal placeholder text "[Surface water / {agency} POTW — confirm discharge pathway]" into an official-looking generated regulatory document, because Site Intake never actually asks the user this. Fix: add a real "Discharge Pathway" field (Surface Water Discharge / POTW-Sanitary Sewer Connection) to the Water-specific section of Site Intake, store it as inputs.dischargePathway, and use it in formConverter.js / documentGenerator.js to print the actual answer instead of a bracketed placeholder.
