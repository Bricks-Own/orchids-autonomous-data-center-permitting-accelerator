BUG: "Update Construction Metrics" (and every other construction save) doesn't actually persist anything. Confirmed by direct API test: PUT /construction/:siteId with percentComplete=50 returns the correctly recomputed data AND "saved": true, but the very next GET/POST returns the generated defaults (percentComplete 76, CPI 1.010) — the user's values are gone.

Root cause, in server/routes.js (~lines 634-660): all three construction endpoints are stateless.
- PUT /construction/:siteId validates and runs calcFullProjectMetrics(updateData), then just returns the result — it never writes anything. "saved: true" is false.
- GET and POST /construction/:siteId call generateConstructionData(...) fresh every time, so whatever the user entered can never come back.
- The frontend compounds it: ConstructionDashboard's handleSaveData discards the PUT response and immediately calls fetchData(), which re-fetches the regenerated defaults — so the user watches their numbers revert right after clicking Save Metrics.

FIX:
1. Add real persistence: a construction_metrics table in server/db.js (site_id TEXT, tenant_id TEXT, data JSON, updated_at) keyed by siteId+tenantId — same pattern as the existing sites table. PUT upserts the validated updateData into it.
2. GET and POST /construction/:siteId must then check for a saved row for that siteId+tenant: if one exists, use the saved user metrics as the base for calcFullProjectMetrics (falling back to generated values only for fields the user never provided); if none exists, generate as today. The precedence rule: user-saved values always win over generated ones.
3. Persist the Vendor Ledger and the Baseline & Projections tab through the same mechanism (their Save buttons currently have the same no-storage problem) — either as additional keys inside the same JSON blob (e.g. data.vendors, data.baseline) or separate rows; blob is fine.
4. Fix the zero-value bug in ConstructionDashboard.jsx handleSaveData: the payload builder uses `formData.X || data?.Y`, so entering 0 in any field (e.g. 0 recordable incidents) silently falls back to the old value. Use ?? (nullish coalescing) or explicit undefined checks instead of ||.
5. After Save, the UI should show the saved values immediately (either use the PUT response directly or re-fetch — both must now agree since the server persists).

Scope guards: tenant-scope the stored rows (WHERE tenant_id = ?) like the sites table; do not touch the dynamic import paths at the top of routes.js (the './lib/...' ones); no Math.random.

VERIFY end-to-end: open Update Construction Metrics, change % Complete to 50 and Actual Cost to 100000000, Save → the C-Suite/Financials KPIs must update to the recomputed values (CPI ≈ 1.058) and STAY that way after a hard page reload and re-login. Then set Recordable Incidents to 0, Save → TRIR must recompute using 0, not the old count. Then edit a vendor row in Vendor Ledger, Save Ledger, reload → the edit survives. Zero console errors.
