ix two bugs in the new Permit Requirements feature on Site Planner
Tested live — the core logic works (toggling a flag or applying a scenario correctly changes which permit sections show on Site Intake), but two display bugs undermine the feature:

Bug 1: Air/Water/Building/Power badges are visually identical regardless of active/inactive state
In SitePlanner.jsx, the badges use variant={p.active ? 'default' : 'outline'}. Check permitOS/src/components/ui/badge.jsx — in this theme, default and outline variants both resolve to the exact same styling (text-foreground, bg-transparent, border-0). There is currently zero visual difference between an active and inactive permit-type badge.

Fix: use variant={p.active ? 'default' : 'secondary'} instead — secondary renders text-muted-foreground in this Badge component, which actually differs from default's text-foreground. Verify by inspecting the rendered class list that active badges are visibly brighter/foreground-colored and inactive badges are visibly dimmer/muted after the change.

Bug 2: The "sent to Intake" confirmation banner shows stale permit types
Reproduced: applying the "Edge Site (Small)" scenario (whose params correctly set hasWaterUse: false) still shows a banner on Site Intake reading "Edge Site (Small) — 60MW — Air, Water, Building, Power" — incorrectly still listing Water, even though the actual applied data (and Step 2's hidden Water section) correctly excludes it.

Root cause: in the scenario card's "Apply → Intake" button and in ScenarioTest's onApply, the code calls previewScenario(scenario) (or handleApplyScenario(params)) — which calls setInputs(...) — and then immediately calls navigateToIntake(...) using activePermitNames, a value computed from inputs during the current render. Since setInputs doesn't update inputs synchronously, activePermitNames in that closure still reflects the site's previous permit flags, not the ones the scenario just applied.

Fix: compute the active permit type names directly from the scenario's own params (or the just-applied params object) at the point of building the summary string, not from the React inputs state variable. For example, in the scenario card handler:

onClick={() => {
  previewScenario(scenario);
  const s = scenario.params;
  const names = [
    s.hasOnSiteGeneration !== false && 'Air',
    s.hasWaterUse !== false && 'Water',
    s.hasNewConstruction !== false && 'Building',
    s.hasGridInterconnection !== false && 'Power',
  ].filter(Boolean).join(', ');
  navigateToIntake(`${scenario.title} — ${scenario.metrics.mw}MW — ${names}`);
}}
Apply the same pattern to the ScenarioTest's onApply handler (use params.hasX directly, not activePermitNames from outer scope) and to the "Send to Site Intake" button (which reads live-toggled inputs directly via the Permit Requirements card, so it may already be correct there since there's no intervening scenario-apply step — but verify it live too, since applyLocation's setInputs call right before it could have the same staleness issue if activePermitNames is read after that call in the same synchronous handler).

Verify after fixing
Toggle a flag on the Permit Requirements card (e.g. Water → No) — confirm the badge in "Current Site Configuration" visibly changes appearance (not just text).
Apply the "Edge Site (Small)" scenario — confirm the banner on Site Intake says "...Air, Building, Power" (no Water), matching the fact that the Water section is correctly hidden on Step 2.
Apply "Hyperscale Campus" (all four flags true) — confirm the banner lists all four.
