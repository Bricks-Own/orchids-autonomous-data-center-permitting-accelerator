Fix Day-1 status colors: several ratio-based metrics show false "CRITICAL" on fresh sites
Confirmed live across multiple tabs on a genuinely fresh site (numbers are correctly zero everywhere, but several status badges incorrectly show red "CRITICAL"):

HSE: "Days Since Last Incident: 0" and "Safety Observations 0/0" both show CRITICAL
Quality: "Inspection Pass Rate: 0%" shows CRITICAL
Financials: "CPI: 0.000" and "Cash Position: $0.0M" both show CRITICAL
Schedule: "SPI: 0.000" shows CRITICAL
The tell that this is fixable narrowly: right next to "SPI: 0.000 — CRITICAL" on the Schedule tab, "Schedule Slip: 0d" correctly shows "N/A" instead of a false alarm. So count/day-based metrics already have sensible zero-handling — this is specifically about ratio-based metrics (CPI, SPI, pass rates, resolution rates, days-since-incident) where getTrafficLight() treats a 0 or 0/0 value as "below every threshold, therefore critical," with no third state for "no data exists yet."

Fix
In getTrafficLight() (server/constructionData.js) and/or the specific KpiCard call sites in ConstructionDashboard.jsx, add an explicit "not yet applicable" check before the normal threshold comparison, for these specific metrics: CPI, SPI, Inspection Pass Rate, Days Since Last Incident, Safety Observations Resolution Rate, TCPI, and any other ratio/rate metric that divides by an accumulated value that's legitimately zero on Day 1.

The rule: if the metric's underlying accumulated denominator is zero (e.g. ACWP === 0 for CPI, PV === 0 for SPI, totalWorkHours === 0 for days-since-incident framing, safetyObservationsTotal === 0 for resolution rate, totalInspections === 0 for pass rate), return a gray/N/A status instead of running the normal green/amber/red threshold logic — exactly the same treatment "Schedule Slip" already correctly gets. Once real activity has been logged (a real "Update Metrics" save with non-zero accumulated values), the normal threshold-based status should take over as it already does today.

Do a full sweep of every KpiCard/TrafficBadge status call across all tabs (HSE, Quality, Financials, Schedule, Other Trackers, PCO Tracker) for the same pattern — any metric whose "good" value only makes sense once real data exists needs this same N/A guard.

What NOT to change
Don't touch the underlying zero-state values themselves — those are correct.
Don't touch the "N/A" handling that's already working correctly (Schedule Slip, Critical Path, Float Consumed).
Don't change the normal (post-Day-1) threshold logic — only add the zero-denominator guard in front of it.
Verify after
Generate a brand-new never-used site — confirm every ratio-based metric across HSE, Quality, Financials, and Schedule shows a neutral "N/A" status, not red "CRITICAL," while the underlying values are still correctly zero.
Save real metrics via "Update Metrics" with non-zero accumulated values — confirm the same metrics now correctly show green/amber/red based on real thresholds, proving the guard only applies to the true zero-data state and doesn't mask real problems once they exist.
