BUG: When the JWT expires (24h expiry, server/auth.js TOKEN_EXPIRY), the app doesn't log the user out — it just silently breaks. Every API call starts failing but the UI stays on the dashboard looking logged-in.

Two root causes:
1. Status-code mismatch: server/middleware.js returns 403 for an invalid/EXPIRED token (line ~58, jwt.verify catch block), but the frontend's session-expiry handling in src/utils/api.js (~line 62) only reacts to 401. So expired tokens never trigger the cleanup path. Fix the semantics server-side: in the jwt.verify catch, distinguish jwt expiry/invalidity and return 401 (401 = "you are not authenticated", which is what an expired token means; keep 403 for authenticated-but-forbidden cases). ALSO make the frontend treat an auth-shaped 403 ("Invalid or expired token") the same as 401, so old deployed backends and the new one both work.
2. No redirect: even on the handled path, api.js just calls clearAuth() and throws — the user stays on a dead dashboard. Fix: when auth expiry is detected in the shared request helper, clear the token AND return the app to the login screen. Cleanest approach given App.jsx's structure: dispatch a window event (e.g. window.dispatchEvent(new Event('auth-expired'))) from api.js; App.jsx listens for it and flips its authenticated state back to the login view, showing a friendly banner on the login form: "Your session expired — please sign in again." (A hard window.location.reload() after clearAuth() is an acceptable simpler fallback since isAuthenticated() will then route to login, but the event approach avoids losing the nicer message.)
3. Proactive expiry (nice-to-have, include it): the JWT payload contains exp. On app load and on an interval (e.g. every 60s), decode the stored token client-side (just base64-decode the payload — no crypto needed) and if exp has passed, trigger the same auth-expired flow immediately instead of waiting for the next failed API call.

Do NOT change TOKEN_EXPIRY itself, the JWT_SECRET handling, or the dynamic './lib/...' import paths in routes.js.

VERIFY: (a) with a valid login, tamper localStorage.permitos_token to a garbage value, click any data-loading tab → app should return to the login screen with the session-expired message, not a dead dashboard; (b) craft a token with a past exp (or temporarily set TOKEN_EXPIRY='5s' locally, log in, wait) → same behavior both on the next API call AND proactively within ~60s without any click; (c) normal login/logout still works; (d) zero console errors beyond the expected handled auth failure.
